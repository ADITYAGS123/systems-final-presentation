🚀 Operating Systems & Embedded Systems

1. Linux Boot: Power-On to Desktop

Basic Flow

Power On
   ↓
BIOS / UEFI
   ↓
POST (Hardware Check)
   ↓
Boot Device Selection
   ↓
Bootloader (GRUB2 )
   ↓
Kernel
   ↓
systemd (init)
   ↓
Services + Desktop

Step-by-Step Explanation

  1. Power On
    CPU starts execution from a fixed memory location (reset vector).
  2. BIOS / UEFI
    Firmware initializes hardware (CPU, RAM, keyboard, etc.).
    • BIOS: Uses MBR (Master Boot Record), limited to ~2TB disks
    • UEFI: Uses GPT, faster boot, supports Secure Boot
  3. POST (Power-On Self Test)
    Checks if hardware components are working properly.
    If any issue is found → error message is shown.
  4. Boot Device Selection
    Firmware selects boot device based on priority:
    Hard disk → USB → CD/DVD (can be configured in BIOS settings)
  5. Bootloader
    Located differently depending on system:
    • BIOS → stored in MBR
    • UEFI → stored as .efi file in EFI partition

    Functions:

    • Locate OS kernel on disk
    • Load kernel into memory
    • Start kernel execution

    Types:

    • GRUB2: Modern, widely used, supports multiple OS
    • LILO: Older, mostly obsolete
  6. Kernel
    After GRUB loads it:
    • Decompresses itself into memory
    • Loads initramfs (temporary root filesystem)
    • Initializes hardware
    • Loads device drivers & kernel modules
    • Starts the init process (systemd (PID 1)) which then starts background processes and everything else
  7. Init Process (systemd)
    First user-space process (PID 1).
    • Handles system startup
    • Mounts file systems
    • Starts all services
  8. Services & System Initialization
    systemd starts:
    • Networking
    • Sound
    • Power management
    • Other background services
  9. Login & Desktop
    • User login (GUI or terminal)
    • Desktop environment loads (GNOME/KDE etc.)

BIOS vs UEFI (Quick Comparison)

BIOS

  • Older system
  • Uses MBR partition
  • Limited disk size (~2TB)
  • Slower boot
  • Less secure

UEFI

  • Modern replacement
  • Uses GPT partition
  • Supports large disks
  • Faster boot
  • Secure Boot support

Key Insight

Booting is a chain process:
Each stage prepares and hands control to the next stage until the system becomes usable.

Exam Tip

Always write this order:
Power → BIOS/UEFI → POST → Bootloader → Kernel → systemd → Services → Desktop

2. Program Execution Process (C/C++)

Basic Flow

Source Code
   ↓
Compiler (Generates Assembly Code)
   ↓
Assembler (Generates Machine Code)
   ↓
Linker (Links Libraries)
   ↓
Loader (Loads into Memory)
   ↓
Execution

Step-by-Step Explanation

  1. Source Code
    A program written in a high-level language like C/C++ that instructs the CPU to perform tasks.
  2. Compiler
    Translates high-level code into assembly language.
    • Checks syntax and semantic errors
    • Performs lexical analysis and syntax checking
    • Ensures program follows language rules
  3. Assembler
    Converts assembly code into machine (binary) code.
    • Output is an object file (.o)
    • Assigns memory locations to instructions
    • Uses physical or virtual addressing
  4. Linker
    Combines object files and libraries to create final executable.
    • Links system libraries and external modules
    • Resolves references (functions, variables)
    • Assigns final memory addresses
  5. Loader
    Loads executable into memory.
    • Allocates memory (code, data, stack, heap)
    • Initializes registers
    • Sets program counter to entry point
    • Starts program execution
  6. Execution
    CPU starts executing instructions and produces output.

Detailed Execution Flow

Write Code → Edit Source Program
        ↓
     Compile
        ↓
   Syntax Errors? ── Yes → Fix Code
        ↓ No
   Object Code Generated
        ↓
 Link with System Libraries
        ↓
 Execute Object Code
        ↓
 Logic/Data Errors? ── Yes → Debug
        ↓ No
   Correct Output → STOP

Key Concepts

  • Object File: Binary representation of program after assembly
  • Libraries: Pre-written code reused during linking
  • Errors:
    • Syntax Errors → during compilation
    • Logic Errors → during execution
    • Data Errors → incorrect input/output
Key Insight:
Compilation converts code → Linking prepares it → Loading runs it.

Exam Tip

Write in order:
Source → Compiler → Assembler → Linker → Loader → Execution

3. Embedded Systems — Purpose-Built Computing

What is an Embedded System?

An embedded system is a combination of hardware and software designed to perform a specific function within a larger system. Unlike general-purpose computers, it is optimized for performance, power, and cost.

Basic Structure

Input Devices (Sensors)
        ↓
Microcontroller / Processor
        ↓
Memory (RAM / ROM / Flash)
        ↓
Output Devices (Actuators / Display)

Explanation of Components

1. Input Devices (Sensors)

Sensors are used to collect data from the environment and provide it to the system. They convert physical signals into electrical signals.

  • Examples: Temperature sensor, pressure sensor, light sensor
  • They act as the input layer of the system

2. Microcontroller / Processor

This is the brain of the embedded system. It processes input data, executes instructions, and controls output devices.

  • Microcontroller (MCU): Used in small, low-power systems
  • Microprocessor (MPU): Used in more complex systems (e.g., embedded Linux)

3. Memory (RAM / ROM / Flash)

Memory stores both program instructions and data required for execution.

  • RAM: Temporary storage during execution
  • ROM: Permanent storage of firmware
  • Flash: Reprogrammable non-volatile memory

4. Output Devices (Actuators / Display)

Output devices convert processed data into physical action or visible output.

  • Examples: Motors, LEDs, displays, buzzers
  • They represent the output layer of the system

Typical Software Stack

Application Code
↓
RTOS / Embedded Linux (optional)
↓
Device Drivers
↓
Hardware (MCU / MPU)

5. Application Code

This is the user-defined program that controls the system behavior and logic. It defines how inputs are processed and what outputs should be generated.

6. RTOS / Embedded Linux

Provides an environment to manage tasks and system resources.

  • RTOS: Ensures real-time execution with strict timing constraints
  • Embedded Linux: Used in complex systems with multitasking support

7. Device Drivers

Drivers act as an interface between software and hardware components.

  • Control hardware like sensors, displays, communication modules
  • Allow application code to interact with hardware easily

8. Hardware (MCU / MPU)

The physical components of the system including processor, memory, and peripherals.

  • Includes GPIO, UART, SPI, I2C interfaces
  • Responsible for actual execution of instructions
Key Insight:
Embedded systems follow a layered approach — hardware at the bottom, software at the top, and drivers/OS connecting both.

Key Characteristics

  • Dedicated to a specific task
  • Real-time operation (in many cases)
  • Low power consumption
  • Compact and efficient
  • High reliability

Types of Embedded Systems

  • Standalone (e.g., microwave, washing machine)
  • Real-Time (e.g., airbag system, pacemaker)
  • Networked / IoT (e.g., smart home devices)
  • Mobile Embedded (e.g., smartphones)

Examples

  • Automotive systems (ABS, engine control)
  • Consumer electronics (TV, camera)
  • Industrial machines
  • Medical devices
  • Routers and IoT devices

Embedded vs General-Purpose Systems

Embedded Systems

  • Single dedicated task
  • Limited resources
  • Real-time constraints
  • Optimized for efficiency

General-Purpose Systems

  • Multiple tasks
  • High processing power
  • No strict timing constraints
  • User-driven applications

Key Components

  • Microcontroller (MCU) – brain of the system
  • Memory – stores program and data
  • Input/Output Interfaces – sensors & actuators
  • Communication Interfaces – UART, SPI, I2C
Key Insight:
Embedded systems are everywhere — from your phone to your car — quietly performing dedicated tasks efficiently.
